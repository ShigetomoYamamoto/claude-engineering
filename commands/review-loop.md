レビューと修正のループを自動実行します。
**サブエージェントは毎回真っ新で起動するため、親(オーケストレータ)が引き継ぎコンテキストを保持してプリアンブルとして渡します。**

# 初回セットアップ(ループ開始前に1回だけ実施)

親セッションが以下を抽出・記憶してください。これらは**ループ中ずっと使い回す**変数として扱います:

- **SCOPE**: 対象範囲(ファイル/ディレクトリ/PR差分など)。ユーザがコマンド起動前に「○○ をレビューして」と指定していればそれを採用。曖昧なら起動直後に「対象範囲を教えてください」と確認
- **FOCUS**: 重点観点があれば(例:「セキュリティ重点」「型安全のみ」)。無ければ reviewer.md のデフォルト観点
- **REVIEW_HISTORY**: 空の配列で初期化(各イテレーションのレビュー結果を蓄積)
- **FIX_HISTORY**: 空の配列で初期化(各イテレーションで触ったファイル+修正概要を蓄積)
- **INSPECTED_FILES**: 空の Set で初期化(reviewer が確認したファイルパスを累積)

# 実行ループ

最大イテレーション数: `$ARGUMENTS`(未指定の場合は 5)。これを **MAX** とします。
イテレーションカウンタを **N**(1 から始まる)とします。

以下のステップを N=1..MAX で繰り返してください:

## ステップ1: レビュー実行

Task ツールで `reviewer` サブエージェントを起動。プロンプトの**先頭に必ず以下のコンテキスト・プリアンブルを置く**:

```
# 引き継ぎコンテキスト(親セッションが管理)

- 現在のイテレーション: N / MAX
- 対象範囲(SCOPE): <SCOPE の内容>
- 重点観点(FOCUS): <FOCUS の内容、無ければ「デフォルト」>
- これまで確認済みのファイル: <INSPECTED_FILES の一覧、初回は「なし」>
- 前回のレビュー指摘: <REVIEW_HISTORY[-1]、初回は「なし」>
- 前回の修正内容: <FIX_HISTORY[-1]、初回は「なし」>

# 今回のタスク
上記コンテキストを踏まえ、SCOPE 配下を再レビューしてください。
特に「前回の修正で本当に直ったか」「修正によって新たに生じた問題は無いか」を重点確認。
既存の確認済みファイルは再読込み不要(構造把握済みとして扱う)。
出力フォーマットは reviewer.md の定義どおり。
```

reviewer の出力を `CURRENT_REVIEW` として保持。
reviewer が確認したファイルパスを `INSPECTED_FILES` に追加(reviewer の出力に出てくる `[ファイル名:行番号]` から抽出)。

## ステップ2: 終了判定

- `CURRENT_REVIEW` が `NO_ISSUES` のみ → ✅ 成功として終了(最終報告へ)
- 何らかの指摘が返ってきた → ステップ3へ

## ステップ3: 重複指摘の検知

`N >= 2` かつ `CURRENT_REVIEW` と `REVIEW_HISTORY[-1]` を比較し、本質的に同じ指摘が2回連続で出ている場合 → 「修正困難な指摘あり」として停止(最終報告へ)

## ステップ4: 修正実行

`REVIEW_HISTORY` に `CURRENT_REVIEW` を追加してから、Task ツールで `fixer` サブエージェントを起動。プロンプトの**先頭に以下のプリアンブル**:

```
# 引き継ぎコンテキスト(親セッションが管理)

- 現在のイテレーション: N / MAX
- 対象範囲(SCOPE): <SCOPE の内容>
- これまでの修正履歴: <FIX_HISTORY の全件、初回は「なし」>

# 今回のタスク
以下のレビュー指摘を 1 件ずつ確実に修正してください。
SCOPE 配下のファイルのみ対象。それ以外は触らない。
過去の修正で既に対処済みの内容が指摘に含まれていれば、その旨を「対応不要(既対処)」として明示。

# レビュー指摘
<CURRENT_REVIEW の全文>
```

fixer の出力(修正したファイル + 概要)を `FIX_HISTORY` に追加。

## ステップ5: 進捗報告

```
[N回目] 指摘 X 件 → 修正完了 (累積修正ファイル: Y 件)
```

## ステップ6: ループ継続

N をインクリメントしてステップ1へ。

# 終了条件まとめ
- ✅ `NO_ISSUES` を受け取った → 成功終了
- ⚠️ 最大イテレーション数に到達 → 残った指摘を提示して停止
- ⚠️ 同じ指摘が2回連続 → 修正困難として停止し、ユーザに判断を仰ぐ

# 最終報告フォーマット

```
## レビューループ完了

- 結果: ✅ 成功 / ⚠️ 未完了
- 対象範囲: <SCOPE>
- イテレーション数: N / MAX
- 累積指摘数: <REVIEW_HISTORY 全件の指摘合計>
- 累積修正ファイル数: <FIX_HISTORY のユニークファイル数>
- 終了理由: NO_ISSUES / 最大回数到達 / 重複指摘検知

(未完了の場合は残った指摘事項を列挙)
```

# 注意
- レビュアーはコードを修正しない、フィクサーは指摘されていない部分を変更しない、という分業を厳守させてください
- サブエージェント呼び出しは独立コンテキストで動くため、必要な情報(SCOPE、前回指摘、前回修正、確認済みファイル)は**毎回明示的に Task の prompt 先頭に置く**こと
- 引き継ぎコンテキストが肥大化したら(目安: 累積 3000 文字超)、要約して詰める。情報の鮮度は「直前 1-2 イテレーション」が最重要、それ以前は要約で OK
