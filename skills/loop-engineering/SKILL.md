# loop-engineering — 検証付きの自律ループで「誰でも」作る/直す

このスキルは、プログラミングが初めての人でも **クローズドループ(検証付き自律ループ)** を回せるようにする案内役です。あなた(Claude)は、利用者に専門知識がない前提で、噛み砕いて・順番どおりに・確認を取りながら進めてください。

> **このスキルが起動した時点で「やってほしい」という依頼は受領済み**です(answer-only 方針に抵触しません)。VISION 作成・テスト追加・実装の編集・検証コマンドの実行はこの範囲で行ってよい。
> ただし **破壊的な操作(git commit/push・マージ・既存ファイルの大量削除・DB/本番への変更・デプロイ・`package.json` 等の設定変更・依存追加)は、その都度かならず利用者に確認**してから行います。勝手に進めないこと。

---

## 全体像(STEP 0 で強さを選んでから 1〜6)

```
[STEP 0] 適用判断(A/B/C)          … タスクの大きさで フル/ミニ/省略 を自動で選ぶ(★最初に必ず)
[STEP 1] 目標を聞き出す            … 何を作る/直すのかを具体化
[STEP 2] VISION.md を書く          … 合格条件を機械でも ○× 判定できる述語で列挙(★核心)
[STEP 3] テスト観点を洗い出す       … VISION の各行 = テスト1個 に展開(同じ ID で貫通)
[STEP 4] レッド→グリーン           … 先に赤を出す → 最小実装で緑 → npm run verify が exit 0
[STEP 5] 作る↔確認ループ           … 「指摘→修正」の往復は /review-loop コマンドに委譲(自作しない)
[STEP 6] 完了判定(Stop condition) … verify が exit 0 + VISION の全[機械]条件が緑 + reviewer が NO_ISSUES
```

**重複しない設計(重要):** STEP 5 の「指摘→修正の往復」は **このスキルで作り直さない**。既存の **`/review-loop` コマンド**に委譲する。ただし **「VISION の各条件が満たされたか」「`npm run verify` が exit 0 か」の判定は委譲先がやってくれない**(reviewer はコード品質しか見ない)。そこは **loop-engineering 自身が STEP 4 / STEP 6 で担保**する。

各ステップ末には **「✅ これが出たら次へ」** を置く。迷ったらそこで利用者に確認すること。

---

## STEP 0: 適用判断(まず”どの強さでやるか”を決める)★最初に必ず

タスクを受け取ったら、いきなり VISION を書き始めず、まず **強さ(A/B/C)を判定**し、選んだ理由を1行で利用者に伝える。「実装して」「直して」等で起動しても、**毎回フルでやらない**ためのゲート。

> **上流から scope を渡されたら再判定しない(入口は1つ・判断者は1人)。** `/autorun` の
> tdd フェーズとして起動された場合(RUN_STATE あり)は、強さ・scope を既に上流(autorun の
> 分析/計画)が決めている。そのときは STEP0 で A/B/C を**判定し直さず、渡された scope を
> 採用**してそのまま STEP2 以降へ進む(同じ判断を二重に持たない)。STEP0 の自前判定は
> **単体起動時(RUN_STATE なし)のみ**行う。根拠: `~/.claude/rules/loop-safety.md`「Single entry, single judge」/ ADR-014。
>
> **逆に、単体起動で「要件・設計の確定が要る/多段の出荷(PR・デプロイ)まで要る」と判明したら、
> コード段に無理に押し込めず `/autorun` に格上げする。** STEP0 は “サイズを測って正しい段(入口)に
> 振り分ける” ゲートでもある(入口は1つ=調査して段を選ぶ。ADR-014)。
>
> **モデル役割ガード注記(ADR-016/ADR-020):** STEP4 以降の実装(編集・変更系 Bash)は「実行」。
> アクティブモデルが**思考ティア(Fable/Opus)**だと `opus-execution-guard` がブロックする(`rules/role-separation.md`)。
> 実装は Sonnet で行う — `/model sonnet` に切り替えるか、編集を Sonnet サブエージェント
> (`fixer` / `tdd-guide`)に委譲する。`/autorun` の tdd フェーズでメインが思考ティアのときも
> 同様に Sonnet 実行へ委譲する。

- **A:フルループ** — 新機能・バグ修正など **観測可能な振る舞いが変わり、テストを置ける**変更。→ STEP 1〜6 を全部やる。
  - **A-中小(1〜数ファイル)= 既定**。計画も実装もメイン会話でインライン。**分離はしない**(ハンドオフで意図が欠落し往復が増えるだけ)。
  - **A-大(多ファイル/移行/仕様が重く 1 コンテキストに載らない)= 規模が要求するときだけ**。計画を Plan エージェント(PMロール)に分離し、人間承認ゲート + workflow `loop-engineering-large-A` に乗せる。詳細は **[reference/large-A.md](reference/large-A.md)**。
- **B:ミニループ** — 小さい変更(小分岐の追加・軽い修正など)。→ VISION.md ファイルは作らず、**述語を1〜2個その場で宣言 → テスト1個 → 実装 → `npm run verify` で exit 0** まで。`/review-loop` はリスクや差分が大きいときだけ。完了判定は簡略(verify 緑＋宣言した述語が緑)。
- **C:省略(そのまま実装)** — 次のいずれか。普通に実装し、**「ループは省略しました(理由: ○○)」と一言添える**:
  - 些末(typo・リネーム・コメント・ログ・整形・1行設定)
  - 使い捨て/お試し/調査(「ちょっと試して」「叩き台」)
  - 非コード(ドキュメント・設定・インフラ)や、**テストが原理的に作れない/テスト基盤が無い**

**判定のコツ:**
- **迷ったら一段下げる**(A↔B 迷い→B、B↔C 迷い→C)。重さは後からいつでも上げられる。
- ただし **データ破壊・課金・認証・セキュリティ・お金や個人情報**に関わる変更は、小さく見えても **A に上げる**。
- 利用者の言葉で動的に変える:「ちゃんとやって」「テストも」→格上げ / 「軽くでいい」「試すだけ」→格下げ。

**✅ これが出たら次へ:** A/B/C を決め、理由を1行で伝えた。A → STEP 1 へ / B → STEP 2(簡略)へ / C → そのまま実装して省略を明記。

## STEP 1: 目標を聞き出す(具体化する)

初心者は「何を伝えればいいか」が分からない。**あなたから順に質問**する(一度に全部聞かず 1〜3 問ずつ)。

**聞くこと(テンプレ):**
1. **何を作りたい/直したいですか?**(例:「保存ボタンを押すと入力が消えるバグを直したい」)
2. **うまくいったと言えるのはどんな状態ですか?**(例:「保存後も入力が残る」)
3. **逆に、こうなったら失敗、という状態は?**(例:「空のまま保存できてしまう」)
4. **対象のファイル/フォルダはどこですか?**(分からなければ「一緒に探します」と言って Glob/Grep で探す)
5. **使えるチェックコマンドはありますか?**(`npm run verify` / `npm test` / `npm run lint` / `npm run typecheck` 等。分からなければ自分で `package.json` 等を読んで調べる)

> **答えられないときの逃げ道:**「では似た例を 2〜3 個こちらから挙げます。近いものを選んでください」と **選択肢を出す**。ゼロから言わせない。
> **頭が整理できていないなら:** 先に `3-line-contract` スキルで目的/出力/成功条件を 3 行にしてから戻ると VISION がぶれない。

**プロジェクトの土台を自分で調べる(初心者の代わりに):** `package.json` / `pyproject.toml` / `Makefile` 等を読み、`test` / `lint` / `typecheck` / それらをまとめた `verify` があるかを把握する。**設定ファイルは勝手に書き換えない**(追加は提案 → 承認後)。

**✅ これが出たら次へ:** 「作る/直す対象」「成功イメージ」「失敗イメージ」が一文ずつ言葉になり、対象の場所と使える検証コマンド(または個別ゲート)が分かった。

## STEP 2: VISION.md を書く ★このスキルの核心

**VISION.md = 「何ができたら成功か」を観測可能な述語だけで書いた合格条件書。** これがループの「ゴールテープ」。これが無いとループは終われない。時間をかける。

> **上流の受け入れ基準を種にする(自走時・受け渡し契約):** `/autorun` の下で動くときは VISION を
> ゼロから作らない。要件段(`requirements-analyst`)/タスク分析段(`task-analyst`)が出した
> **受け入れ基準(テスト可能な形)** を入力に取り、各基準を1つの機械判定述語に変換する(同じ ID を
> 引き継ぐ)。これで「人間が関門で承認したもの = 最下段で機械が検証するもの」が一致し、ループが
> 上段から下段まで閉じる。`~/.claude/docs/autorun-flow.md`「Requirement → VISION handoff」/ ADR-014。

### 良い述語 / 悪い述語

| 悪い(測れない) | 良い(観測可能な述語) |
|---|---|
| 入力を使いやすくする | `N1 [機械]` 有効なメールで送信すると 200 と `{ok:true}` が返る |
| エラーをちゃんと出す | `E1 [機械]` メール欄が空で送信すると 400 と `{error:"required"}` が返る |
| 変な値に強くする | `B1 [機械]` 上限255字ちょうどは成功し、256字は 400 になる |
| ログイン状態を保つ | `S1 [機械]` ログイン後にリロードしても認証済み画面が表示される |
| 見た目を整える | `Q1 [AI]` 送信ボタンは無効時にグレーアウトし、押下不可と分かる |

コツは **「○○すると △△ になる」** の形で、**外から見て確かめられる結果**を書くこと。

> **述語は非エンジニアが読んで分かる言葉で書く。** 利用者が画面で見る/操作する内容で表す。**プロジェクト固有の名前(store・状態管理・関数名・内部変数・`origin=user` のような内部値)は述語に入れない** → それらは「テスト方針」(どう assert するか)側に書いて保持する(情報は捨てない)。`textarea`・クリック・フォーカス・`blur`・Enter のような**どのアプリでも通じる汎用UI用語は使ってよい**。例: `保存→解除＋store更新(origin=user)`(✗ 内部が混入)→ `保存すると編集が終わり元の表示に戻る`(○ 述語)＋ テスト方針に「保存時 store が origin=user で更新される」。

### 各条件に振る 2 つのラベル

1. **ID**(カテゴリ頭文字+連番)。VISION → 観点表 → テスト → 完了判定まで **同じ ID で一気通貫**させる。
   - `N` = 正常系(Normal) / `E` = 異常系・エラー処理(Error。例外時の振る舞いも E に含める) / `B` = 境界値(Boundary) / `S` = 状態(State)
   - (任意)`Q` = 見た目・文言など曖昧な品質
2. **検証手段タグ**(誰が ○× を出すか)。
   - `[機械]` = テスト/lint/型チェックで自動判定できる(**最優先で土台にする**)
   - `[AI]` = 機械で原理的に測れないものだけ AI に採点させる(**乱用しない**)
   - **迷ったら必ず `[機械]` に寄せる。**

### VISION.md テンプレ

そのままコピーして埋める雛形は **[reference/vision-template.md](reference/vision-template.md)** に置いた(完了条件節 + ID/検証手段タグ付きの合格条件表 + スコープ外)。

> **初心者がつまずく所:**「全部の条件を最初から完璧に出さなきゃ」と固まる。→ 不要。**正常系1・異常系1・境界1 の3行だけでもまず書けば回り始める**。後から足せる。最初の1行は一緒に書いてあげる。
> **保存と承認:** プロジェクトの分かりやすい場所(`VISION.md` or `docs/VISION-<名前>.md`)に置く。新規作成は破壊的ではない。既存の同名があれば上書き確認。作ったら**全文を見せて「この条件でOK?抜けは?」と承認をもらう**。

**✅ これが出たら次へ:** ID + 検証手段タグ付きの合格条件が最低3行あり、各行が「○○すると △△ になる」の機械判定できる言い切りで、完了条件節が埋まり、利用者が承認した。

## STEP 3: テスト観点を洗い出す(VISION → テストへ展開)

VISION の各行を **テスト1個**に落とす。**ID は VISION と同じものを使い回す**(N1 の条件 → N1 のテスト)。これで「条件 ↔ 観点表 ↔ テスト」が同じ番号で一直線になり、抜け漏れと迷子を防ぐ。

### 5 つの軸(出発点。完全網羅ではない)

各入力にこの軸を順に当てて「テストすべき状況」を機械的に出す:

| 軸 | 接頭辞 | 何を問うか | 例(メール入力) |
|----|--------|-----------|----------------|
| 正常系 | N | 普通の入力で期待どおり動くか | 正しいメール → 成功 |
| 異常系 | E | 不正・想定外で安全に失敗するか | 空 / 形式不正 / 型違い → 期待エラー |
| 境界値 | B | 端っこで壊れないか | 255字 OK / 256字 NG、0件 / 1件 |
| 状態 | S | 前後の状態・連続操作・永続化は | 連続送信、リロード後、重複登録 |
| エラー処理 | E | 失敗が握り潰されず伝わるか | 依存先ダウン時に 500 でなく 503+案内 |

> 4〜5 軸は「考え始める足場」であって完全網羅の保証ではない。各入力に当てた後、「この軸では拾えない固有のリスクは?」を一度だけ自問して追加する。1条件がテスト1個に収まらなければ `N1-a` `N1-b` と枝番で割る。

### テストの置き場所を確認

既存テスト(`*.test.ts` / `*_test.py` / `tests/` 等)を Glob で探し、そのプロジェクトの流儀に合わせる。テスト実行コマンドを確定する。テストが1つも無いなら最小の1ファイルから(フレームワークは既存依存から判断、無ければ確認)。

**✅ これが出たら次へ:** VISION の各 ID に対応するテスト方針(どのファイルに何を assert するか)が決まり、テスト実行コマンドが確定した。

## STEP 4: レッド → グリーン(ここはこのスキルが直接やる)

**ユーザーのグローバル方針(testing.md)に厳密に従う:レッド → グリーン → リファクタ。**

### 機能追加
1. 観点表の `[機械]` 行を、**実装より先に失敗するテスト**として書く。
2. 実行して **赤(失敗)になることを目視**する。赤くならないテストは検出力ゼロ(偽パス)→ 書き直す。
3. 通る **最小限の実装**を書く(グリーン)。緑のまま整える(リファクタ)。

### バグ修正
1. **バグを再現する失敗テストを先に書く**(レッド)= バグの述語化そのもの。
2. 赤を確認 → 最小修正でグリーン。

### 後付けテストになった場合(ミューテーションで事後証明)
実装が先行した/既存実装にテストだけ足す場合は、検出力を事後証明する:
1. テストが守るべき実装の重要行を**わざと壊す**(削除/条件無効化/分岐を固定値に)。
2. 対象テストだけ実行し、**赤くなることを確認**(`npx vitest run <file>` 等)。
3. 実装を**元に戻す**(`git checkout -- <file>`)。
4. 再び緑に戻ることを確認。
5. 赤くならなければ偽パス。同じミューテーションで赤くなるよう作り直す。

> なぜ赤を先に見るのか:**テスト自身がちゃんと働いている証拠を先に取る**ため。赤を一度も見ていないテストは「実装に合わせて書いただけ」で回帰を検出できない飾りになりがち。

仕上げに **`npm run verify`(typecheck && lint && test 相当)を実行し exit 0** を確認する。まとめコマンドが無ければ typecheck → lint → test を個別実行し全部 exit 0。

**✅ これが出たら次へ:** 最初に書いたテストが全部緑で、`npm run verify` が exit 0。

## 検証ゲートは 2 種類(役割を間違えない)★中核

| | 機械的検証 | AI 検証 |
|---|---|---|
| 中身 | テスト / lint / 型チェック | LLM に見た目・文言・トーンを採点させる |
| 性質 | **決定的・安い・回帰に強い** | 非決定的・ブレる・高コスト |
| 判定 | **exit code 0 / 非0** で自動 | 合否/スコアを文章で |
| 位置づけ | **土台(必須)** | **上乗せ(曖昧な品質だけ)** |

原則:**土台は機械的検証**。AI 検証は VISION の `[AI]` 行だけに使い、機械で測れるものを AI 任せにしない。

---

## STEP 5: 作る↔確認ループ(/review-loop コマンドに委譲。作り直さない)★中核

STEP 4 で「最初のテスト+実装が緑、`npm run verify` が exit 0」になったら、ここからの **「厳しくレビュー → 直す → 再レビューを指摘0まで」の往復**は、**自前で実装せず `/review-loop` コマンドに委譲**する。これが maker/checker 分離(作る人に自己採点させない)の実体になる。

### /review-loop の正体と渡し方

- 実体は `~/.claude/commands/review-loop.md` の **スラッシュコマンド**(スキルではない。`Skill(review-loop)` では呼べない)。`/review-loop [最大回数]`(既定 5)で起動する。
- このコマンドは **親オーケストレータが SCOPE/FOCUS/履歴を保持し、reviewer → fixer を Task で指摘0まで回す**手順。**loop-engineering 自身がその親**になる(中身が見えないブラックボックスを呼ぶのではない)。
- 起動前に決めておくもの:
  - **SCOPE** = レビュー対象(STEP 1 で決めたファイル/フォルダ/差分)。
  - **FOCUS** = 重点観点(任意。例「セキュリティ重点」)。無ければ reviewer のデフォルト観点。
  - **最大回数** = VISION の往復上限(既定 5)。**コマンド引数で渡せるのはこれだけ。**
- ⚠️ **VISION.md や完了条件(verify exit 0)はコマンドに渡せない**(受け口が無い)。それらは loop-engineering 側が保持し、**STEP 6 で自分で照合**する。reviewer 起動時の FOCUS や根拠としてプリアンブルに注入はできる。

### クロスモデル版(任意・別モデルでレビューさせたいとき)

- `/review-loop-cross [最大回数]`:別モデル(Codex/GPT)がレビュー。**SCOPE は未コミットの git 差分に固定** → 非 git や差分が無い場合は使えない。
- `/review-loop-cross-path <path> [最大回数]`:git 管理外 OK。**`<path>` 必須・既定上限 2**(他と違う)。5 回回したいなら明示的に渡す。
- ⚠️ クロス版は**外部 CLI(codex)を実行し、ネットワーク通信・外部サービス利用量が発生する副作用**がある。使う前に利用者へ確認(answer-only の精神)。

### reviewer / fixer の実際の挙動(/review-loop が中で回す。再定義しない)

- **reviewer(確認担当 / checker):** バグ・並行性・セキュリティ・パフォーマンス・設計の**純粋なコードレビュー**。出力は機械可読で、**CRITICAL/HIGH のみ列挙、無ければ `NO_ISSUES` の1行**だけ。**`npm run verify` や exit code や VISION は見ない**(だから verify と VISION 照合は loop-engineering の仕事)。
- **fixer(実装担当 / maker):** 指摘を **1 件ずつ最小修正**。バグはレッド→グリーン、後付けはミューテーションで検出力確認、修正後に **`npm run verify` で exit 0** を確認。**誤った指摘には従わず、理由を添えて報告**する。
- `/review-loop` は **「同じ指摘が2回連続なら停止」「履歴が肥大したら要約」** を内蔵している。loop-engineering は二重に実装せず、**結果(NO_ISSUES か、上限/重複で停止か)を受け取るだけ**。

**✅ これが出たら次へ:** `/review-loop` が `NO_ISSUES` で戻った(または上限/重複で停止 → STEP 6 のエスカレーション)。

---

## STEP 6: 完了判定(Stop condition)★中核

`/review-loop` が NO_ISSUES で戻っても、**「VISION の全条件が満たされたか」「verify が exit 0 か」は誰も担保していない**(reviewer はコード品質しか見ない)。だから完了判定は **loop-engineering 自身が次の順で行う**(この順番は reviewer の仕様ではなく**このスキルのポリシー**):

1. **まず機械的ゲート:** `npm run verify` の exit code。**0 以外なら未完** → STEP 4 へ戻る(壊れているコードの見た目を褒めても無意味)。
2. **緑のときだけ、VISION の各 `[機械]` 条件に対応するテストが存在し緑かを同一 ID で照合。**
3. **最後に `[AI]` 条件だけ**、見た目/文言を AI に採点させる。

### 完了条件(これが全部 ○ なら「完成」)

- [ ] `npm run verify`(typecheck && lint && test 相当)の **exit code が 0**(= 単一述語に集約。fixer/reviewer が見る機械ゲートと同一)
- [ ] VISION の全 `[機械]` 条件に対応するテストが緑(ID 照合)
- [ ] `reviewer` が `NO_ISSUES`
- [ ] (`[AI]` 条件がある場合のみ)AI 検証もすべて pass

### 完了条件を勝手に緩めない(独立ルール)
「テストを消して緑にする」「条件を削る」「上限を黙って増やす」は**禁止**。詰まったら相談する。

### 検証コマンドが無いとき
`npm run verify` 等が無いリポジトリでは、勝手に設定を書き換えず、以下のいずれかを確認・提案する:
- 既存の `npm test` / `pytest` / `go test` 等を機械的ゲートに採用する。
- `verify` 相当(typecheck && lint && test を1コマンドに)の追加を**提案**し、承認後に作る。
- それも無ければ、最低限「対象テストを実行して exit 0」を暫定ゲートにし、その旨を明記する。

### 暴走防止(かならず守る)
- **往復上限(既定 5、cross-path は 2)。** 超過、または同じ指摘が2回連続したら **止めて人間にエスカレーション**(無限ループで費用を溶かさない安全弁)。
- **破壊的操作は毎回確認:** push / マージ / 削除 / DB・本番変更 / 依存追加(`package.json` 変更)など。
- **完了条件は改竄しない**(上記ルール)。
- **安全規律の正本は `~/.claude/rules/loop-safety.md`。** 本スキルの往復上限(既定 5 / cross-path 2)は loop-safety のハードストップ(ターン/時間/トークン)とは**独立した二重ブレーキ**で、いずれか先に到達した方で停止する。破壊的操作確認・ゴールドリフト・完了条件改竄禁止の規範と数値はそちらに従う。

## やってはいけないこと(アンチパターン)

- ❌ VISION を書かずにいきなり実装する(止め時が無くなる)。
- ❌ 述語が曖昧(「使いやすく」「ちゃんと」)。○× が付かない条件は条件ではない。
- ❌ 機械で測れるものを `[AI]` にする(非決定的になり回帰検知が壊れる)。
- ❌ 赤を一度も見ずにテストを「通った」とする(偽パスの温床)。
- ❌ 機械的ゲートが赤(exit 非0)なのに AI 採点に進む / 見た目を褒める。
- ❌ 同じ AI に実装と採点を両方させる(maker = checker)。
- ❌ `/review-loop` と同じ往復ロジックを自前で書く(委譲する)。
- ❌ `/review-loop` を `Skill(...)` で呼ぼうとする(スラッシュコマンドである)。
- ❌ VISION/verify 照合を reviewer に任せたつもりになる(reviewer はコード品質しか見ない。照合は自分でやる)。
- ❌ 完了条件を満たせないからとテストを消す・条件を削って緑にする。
- ❌ 依頼範囲を超える破壊的操作(commit/push/マージ/削除/DB/本番/依存追加)を独断で行う。
- ❌ 中小規模の A でまで計画を別人格に切る・Workflow を使う(過剰オーケストレーション。ハンドオフで意図が欠落し往復が増える)。
- ❌ 大規模A で計画の人間承認ゲートを飛ばして実装に入る / 赤テストを見ずに別人格へハンドオフする。
- ❌ 実装中に判明した VISION の穴を独断で解釈して埋める(戻りチャネルで承認ゲートに戻す)。
- ❌ レビュー往復(reviewer↔fixer)を workflow の中に自作で組み込む(`/review-loop` に委譲する)。

## クイックリファレンス(毎回守る要点)

0. まず **適用判断(A/B/C)**:些末・使い捨て・テスト不可は **C(省略)**、小変更は **B(ミニ)**、ちゃんとした機能・バグ修正は **A(フル)**。毎回フルを強制しない。A の中でも **中小はインライン / 大規模(多ファイル・移行・重い仕様)だけ** 計画を Plan に分離+人間承認ゲート+workflow `loop-engineering-large-A`。規模が要求しない分離はしない。実装中の VISION の穴は**戻りチャネル**でゲートに戻す。
1. 実装より先に **VISION.md(機械でも ○× 判定できる述語 + ID + 検証手段タグ)** を書く。
2. **VISION の行 = 観点表の行 = テスト1個** を同じ ID で貫通。4〜5 軸(正常/異常/境界/状態/エラー)は出発点。
3. テストは **先に赤を見てから緑**。後付けはミューテーションで検出力を証明。
4. 土台は **機械的検証(exit 0)**。AI 検証は曖昧な品質だけ上乗せ。
5. 「作る↔確認」往復は **`/review-loop` コマンドに委譲**(自作しない)。SCOPE/FOCUS/最大回数を渡す。
6. **VISION 照合と verify は自分(STEP6)でやる**。reviewer は見てくれない。完了 = verify exit 0 + 全[機械]条件が緑 + NO_ISSUES(必要なら AI 検証も)。
7. 往復は **上限 5 回(cross-path は 2)**。超過・同一指摘2連続は **止めて人間に相談**。破壊的操作・クロス版の外部CLIは **毎回確認**。
8. 初心者を一人にしない。各 STEP で「✅ これが出たら次へ」を確認し、詰まったら逃げ道を出す。
